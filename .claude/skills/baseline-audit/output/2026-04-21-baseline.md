### 2. Size prop 對齊

| Checkbox | sm/md/lg | 否(sm=16 / md=20 / lg=24,自訂) | self-contained primitive 特殊比例 |
| Field | sm/md/lg | 是(繼承) | Field wrapper |

**其他**:
- 無 cva size 的 Family 4 元件(Input / NumberInput / DatePicker / Select / Combobox / LinkInput / PeoplePicker)→ size 繼承自 `<Field>` context(非自有 variant)
- 無 cva size 的 composite / primitive(Accordion / Breadcrumb / Dialog / Popover / ... 多數) → 無 size prop 或使用單一固定尺寸
- **Avatar** 用 object map(xs/sm/md/lg/xl,非 cva)—— 見 spec 註記「驅動 inline style variants 要用 object map 非 cva」
- **Chip** 用 cva 但 size 鎖固 `h-field-sm`(spec 明文:Material 3 慣例)
- **Notice / Alert / Toast** 明文「單一固定 size,不實作 sm/md/lg」

**命名偏離**:零(全部 `sm/md/lg` 或 `xs/sm/md/lg`)。無 `small/medium/large`、無 `xxs/xl` 等例外。

### 3. Dismiss pattern(分布)

| Pattern | 元件清單 |
|---------|---------|
| **`onDismiss` callback + 內建 UI**(Tag-style X icon)| Tag, Notice(→ Alert / Toast 繼承) |
| **Tag-style X icon**(consumer 傳 Tag,元件內部用 Tag onDismiss)| Combobox(multi tag), Select(multi tag), PeoplePicker(PersonDisplay 包 Tag) |
| **`onClear` via ItemInlineAction**(Family 4 field)| Input, LinkInput, DatePicker(clear button),NumberInput(邊界狀態) |
| **Radix managed(X close button inside SurfaceHeader / ItemInlineActionButton)**| Dialog, Popover, Sheet, FileViewer(`onClose`) |
| **Button tertiary iconOnly delete**(非 ItemInlineAction)| FileItem(刪除按鈕) |
| **無 dismiss**| 多數 self-contained / composite(AspectRatio / Avatar / Badge / Breadcrumb / Button / Calendar / Carousel / ... ) |

**觀察**:三種 canonical shape 並存:`onDismiss`(Tag/Notice)/ `onClose`(Dialog/Sheet/FileViewer/Popover)/ `onClear`(Field Clear)/ `onRemove`(PeoplePicker)。至少 4 個 prop 名稱在流通。

### 5. Padding token 使用(分布)

| 來源 | 用量(全 DS component tsx 合計) | 元件數 |
|------|--------------------------------|--------|
| **`--layout-space-*`** | 39 refs | 7 元件(DescriptionList x5, Dialog x2, FileViewer x8, Sidebar x13, Tabs x6, TreeView x2) |
| **Tailwind p-N(預設 scale)** | ~60 refs | 30+ 元件(普遍) |
| **任意值 `p-[var(--...)]` / `p-[Xpx]`** | ~60+ refs | 15 元件(Button, Combobox, DescriptionList, Dialog, FileViewer, Menu, ScrollArea, Select, SegmentedControl, SelectionControl, Sidebar, Tabs, TreeView 等) |
| **無 padding**(pure composition / primitive) | — | 其餘 |

**觀察**:
- `layout-space-*` token 集中在**頁面級/section 級 layout**(Sidebar / Tabs / Dialog / FileViewer / DescriptionList / TreeView)
- Tailwind `p-N` 在**元件內部 slot padding**(Button, Menu, Tag, Chip 等)廣泛
- `p-[...]` 任意值多數是 `p-[var(--field-height-sm)]` / `p-[var(--layout-space-*)]` 等 token-backed,只有少量位置硬寫 pixel
- **混用** is the norm,沒有單一 canonical。baseline 記現況,canonical 判斷留 Phase 2

## Methodology note

本輪 baseline 依 SKILL.md 14 面向 + references/dimension-heuristics.md heuristic 掃。特殊處理:

- **Family scope 聲明**:Avatar / AspectRatio / Badge / Checkbox / CircularProgress / FileUpload / ProgressBar / RadioGroup / Rating / ScrollArea / Separator / Skeleton / Slider / Switch 歸 `self-contained primitive`(spec 明文),Accordion / Breadcrumb / Calendar / Carousel / Chart / CheckboxGroup / Coachmark / Command / DataTable / DescriptionList / Dialog / Empty / FileViewer / HoverCard / NameCard / Popover / Sheet / SelectMenu / Tabs / Tooltip / Toast(via Notice)歸 `composite / multi-section`(spec 明文)。
- **Gap/Padding/Icon**:以 `.tsx` primitive 為準,stories 的 anatomy demo 用法(例如 `active ? 'bg-primary...'` 之類的 toggle 示意)不算入 baseline,因為是 stories 展示 code 非元件實作。
- **Tag `text-[var(--color-*)]`**:本質是 color utility 不是 font-size,不算 typography 違規。只有 Badge `text-[10px]`(font-size)算 typography 硬寫。
- **Command / Sheet `text-sm/xs/lg`**:屬 shadcn template 遺留,已記在 matrix typography 欄。

### 面向未擴充清單(本 skill 範圍內無法判斷的項目)

以下是「matrix 欄位無法決定某元件該歸哪類」或「heuristic 對特定案例有雙重解讀」的項目,列給使用者定 canonical(Phase 2 判斷才能進行):

1. **Dismiss pattern canonical 未定義**:`onDismiss` vs `onClose` vs `onClear` vs `onRemove` 四種 callback 名稱並存。是否要統一?由「意圖」(dismiss 提示 vs 關閉浮層 vs 清空欄位 vs 移除成員)分類還是統一?
2. **Badge overlay anchor canonical 未定義**:`badge?` prop 在 Button/Chip/SegmentedControl/Menu/DropdownMenu/Tabs/Sidebar 是 **inline badge**(在 pill 尾端顯示 count),在 Avatar/NameCard 是 **overlay dot**(角標定位)。兩者視覺 / 語意不同,是否要分兩個 prop 名(如 `badge` vs `indicator`)?目前混用同名。
3. **Padding source canonical 未定義**:`layout-space-*` vs `p-N` Tailwind vs `p-[var(--...)]` 任意值三者並存。幾個元件(Sidebar / FileViewer / TreeView / Tabs)同檔混用。是否有**元件內部 padding**(Tailwind)vs **section layout padding**(layout-space token)的明確分界規則?
4. **Icon size 來源 canonical 未定義**:`ICON_SIZE` const(來自 item-anatomy) vs inline `size={16}` vs `w-4 h-4` / `size-4` 三路並存。Sidebar 同時用三種。是否要強制 item-anatomy 內走 const,其他元件走 inline?
5. **Size prop 對齊 field-height 的例外 rationale**:Checkbox / RadioGroup / Switch / Slider / Textarea 的 sm/md/lg 未完全對齊 `field-height-*`(各自有 rationale 在 spec),但**不同元件的 rationale 格式 / 位置不統一**(有的在 spec 第一段,有的藏在後段)。是否要 canonical「偏離 field-height family 必須在 spec rationale 段明寫」的檢查點?
6. **composite 元件的 Family 聲明是否要細分**:目前「composite / multi-section」一詞涵蓋 17 個元件,但其中:
   - 有 `Dialog / Popover / Sheet / Tooltip / HoverCard / Coachmark / Command / SelectMenu`(浮層 composite)
   - 有 `Tabs / Accordion / DataTable / Calendar / Carousel / Breadcrumb`(頁面 composite)
   - 有 `CheckboxGroup`(pure wrapper)
   是否這 17 個該再細分(例如 `overlay-composite` vs `page-composite` vs `layout-wrapper`)?

以上 6 項屬 Phase 2「canonical 判定 + rationale gap」的工作,baseline 不下判斷,只列給使用者看。
